# 개발 계획서

## 1. 문서 개요

- 프로젝트명: TeamSync Manager
- 목적: Streamlit 기반 팀 프로젝트/일정/채팅 관리 웹서비스 개발 계획 수립
- 배포 방식: GitHub Repository 연동 후 Streamlit Cloud 배포
- 기준 문서: `PRD.md`
- 작성일: 2026-06-24

## 2. 개발 목표

본 개발의 목표는 회사 팀 단위로 프로젝트, 일정, 채팅을 통합 관리할 수 있는 MVP 웹서비스를 구현하는 것이다.

MVP는 다음 기능을 우선 완성한다.

- 로그인/로그아웃
- 사용자 및 팀 관리
- 프로젝트 관리
- 태스크 관리
- 근무/출장/휴가 일정 관리
- 팀/프로젝트 채팅
- 관리자 대시보드
- Streamlit Cloud 배포
- DB 없이 샘플 데이터로 동작하는 테스트 버전

## 3. 개발 원칙

- Streamlit에서 안정적으로 운영 가능한 구조를 우선한다.
- MVP에서는 기능을 과도하게 확장하지 않는다.
- 테스트 버전은 DB 없이 JSON 샘플 데이터와 `st.session_state`로 동작하게 만든다.
- DB 접속 정보, 비밀번호, API 키는 GitHub에 올리지 않는다.
- 인증, 권한, 데이터 접근 범위는 초기부터 분리해서 구현한다.
- 화면 코드는 단순하게 유지하고, 저장소/권한/비즈니스 로직은 별도 모듈로 분리한다.
- 저장소 계층은 JSON 기반 테스트 저장소에서 DB 기반 운영 저장소로 교체 가능하게 설계한다.
- 실시간 채팅은 MVP에서 주기적 새로고침 또는 수동 새로고침 방식으로 구현한다.

## 7. 개발 단계

## Phase 0. 프로젝트 초기 세팅

### 목표

Streamlit 앱 개발을 시작할 수 있는 기본 구조를 만든다.

### 작업 목록

- GitHub Repository 생성
- 로컬 프로젝트 폴더 구성
- Python 가상환경 준비
- `requirements.txt` 작성
- `.gitignore` 작성
- Streamlit 기본 실행 확인
- `.streamlit/config.toml` 작성
- README 초안 작성

### 산출물

- 실행 가능한 Streamlit 기본 앱
- GitHub에 push 가능한 기본 프로젝트 구조

### 완료 기준

- `streamlit run app.py` 실행 시 기본 화면이 열린다.
- GitHub에 민감 정보 없이 push 가능하다.

## Phase 1. 테스트 저장소 및 샘플 데이터 구성

### 목표

데이터베이스 없이 앱을 실행할 수 있도록 JSON 샘플 데이터와 저장소 모듈을 완성한다.

### 작업 목록

- `data/` 폴더 생성
- 사용자/팀/프로젝트/태스크/일정/채팅 샘플 JSON 작성
- `src/storage.py` 구현
- 앱 시작 시 JSON 데이터를 `st.session_state`로 로드
- 생성/수정/삭제 작업을 세션 데이터에 반영
- 로컬 테스트용 JSON 저장 함수 구현
- 운영 전환용 `sql/schema.sql`은 후순위로 작성

### 주요 테스트 데이터

- `teams`
- `users`
- `projects`
- `project_members`
- `tasks`
- `schedules`
- `chat_rooms`
- `chat_messages`

### 산출물

- 테스트 JSON 데이터
- JSON/세션 기반 저장소 모듈
- 샘플 관리자 계정

### 완료 기준

- Streamlit 앱이 DB 없이 실행된다.
- 샘플 관리자 계정으로 로그인할 수 있다.
- 샘플 프로젝트, 일정, 채팅 데이터가 화면에 표시된다.

## Phase 2. 인증 및 권한

### 목표

로그인/로그아웃과 사용자 권한 제어를 구현한다.

### 작업 목록

- 테스트용 샘플 계정 방식 결정
- 운영 전환 시 적용할 비밀번호 해시 방식 정리
- 로그인 화면 구현
- 로그인 성공 시 `st.session_state` 저장
- 로그아웃 기능 구현
- 권한별 메뉴 표시
- 로그인하지 않은 사용자 접근 제한
- 권한 체크 함수 구현

### 권한

- `Admin`: 전체 관리
- `Manager`: 소속 팀 관리
- `Member`: 본인 데이터와 참여 프로젝트 관리

### 산출물

- 로그인 화면
- 세션 관리 로직
- 권한 체크 모듈

### 완료 기준

- 올바른 계정으로 로그인할 수 있다.
- 잘못된 계정은 로그인할 수 없다.
- 권한에 따라 표시되는 메뉴가 다르다.

## Phase 3. 사용자 및 팀 관리

### 목표

관리자가 사용자와 팀을 관리할 수 있도록 한다.

### 작업 목록

- 팀 목록 조회
- 팀 생성/수정/삭제
- 사용자 목록 조회
- 사용자 생성/수정/비활성화
- 사용자 팀 배정
- 사용자 권한 변경
- 관리자 화면 구성

### 산출물

- 관리자 페이지의 팀 관리 기능
- 관리자 페이지의 사용자 관리 기능

### 완료 기준

- Admin은 사용자와 팀을 생성/수정할 수 있다.
- Manager/Member는 관리자 기능에 접근할 수 없다.

## Phase 4. 프로젝트 관리

### 목표

팀 단위 프로젝트 생성과 진행 상태 관리를 구현한다.

### 작업 목록

- 프로젝트 목록 화면 구현
- 프로젝트 생성 폼 구현
- 프로젝트 수정 기능 구현
- 프로젝트 삭제 또는 보관 기능 구현
- 프로젝트 상태 변경
- 담당자 지정
- 프로젝트 참여자 지정
- 프로젝트 상세 화면 구현
- 프로젝트 검색/필터 구현

### 산출물

- 프로젝트 목록 화면
- 프로젝트 상세 화면
- 프로젝트 CRUD 기능

### 완료 기준

- 사용자는 권한 범위 내 프로젝트를 조회할 수 있다.
- Manager 이상은 프로젝트를 생성/수정할 수 있다.
- 프로젝트 상태와 마감일을 관리할 수 있다.

### 산출물

- 프로젝트 상세 내 태스크 관리 UI
- 태스크 CRUD 기능

### 완료 기준

- 프로젝트별 태스크를 등록하고 담당자를 지정할 수 있다.
- 담당자는 본인 태스크 상태를 변경할 수 있다.

### 산출물

- 팀/프로젝트별 메시지 저장 기능

- 최근 메시지를 시간순으로 확인할 수 있다.

## 15. 운영 DB 전환 계획

테스트 버전이 안정화된 후 운영 버전으로 전환할 때는 다음 순서로 진행한다.

1. Supabase 또는 Neon PostgreSQL 선택
2. JSON 데이터 모델을 기준으로 DB 스키마 작성
3. `src/storage.py`와 `src/repositories.py`의 내부 구현을 DB 방식으로 교체
4. Streamlit Secrets에 DB 접속 정보 등록
5. 샘플 JSON 데이터 일부를 DB seed 데이터로 이관
6. 권한별 데이터 조회가 동일하게 동작하는지 재검증

운영 DB 전환 전까지 Streamlit Cloud 배포본은 테스트/데모 용도로 사용하며, 여러 사용자가 동시에 입력한 데이터를 영구 보존하는 용도로 사용하지 않는다.
