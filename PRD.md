# 팀 프로젝트/일정/채팅 관리 웹서비스 PRD

## 1. 문서 개요

- 제품명: TeamSync Manager
- 작성일: 2026-06-24
- 배포 환경: Streamlit Community Cloud 또는 Streamlit Cloud 유사 환경
- 소스 관리: GitHub Repository
- 테스트 데이터 저장 방식: DB 없이 JSON 파일 및 `st.session_state` 기반 저장
- 주요 목적: 회사 내 팀 단위로 프로젝트, 업무 일정, 근무/출장/휴가 일정, 팀 채팅을 한 화면에서 관리하는 웹서비스 구축

## 2. 배경 및 문제 정의

회사 내 팀 업무는 프로젝트 관리 도구, 캘린더, 메신저, 엑셀 파일 등으로 분산되어 관리되는 경우가 많다. 이로 인해 다음 문제가 발생한다.

- 팀별 프로젝트 진행 상태를 한눈에 보기 어렵다.
- 근무, 출장, 휴가 등 구성원 일정을 프로젝트 일정과 함께 파악하기 어렵다.
- 프로젝트 관련 대화가 별도 메신저에 흩어져 나중에 찾기 어렵다.
- 관리자는 팀별 리소스 현황과 일정 충돌을 빠르게 확인하기 어렵다.

본 서비스는 팀 단위 업무 관리를 위한 간단하고 실용적인 내부 웹서비스를 목표로 한다.

## 3. 목표

### 3.1 제품 목표

- 팀별 프로젝트와 업무 일정을 통합 관리한다.
- 개인 및 팀 단위의 근무, 출장, 휴가 일정을 등록하고 조회한다.
- 프로젝트별 또는 팀별 채팅을 제공해 업무 맥락을 보존한다.
- Streamlit 기반으로 빠르게 개발하고 GitHub 연동 배포가 가능하도록 한다.

### 3.2 성공 기준

- 사용자는 로그인 후 본인이 속한 팀의 프로젝트, 일정, 채팅을 확인할 수 있다.
- 관리자는 팀원, 프로젝트, 일정 유형을 관리할 수 있다.
- 프로젝트별 진행 상태와 마감일을 대시보드에서 확인할 수 있다.
- 일정 중복, 휴가자, 출장자를 주간/월간 단위로 파악할 수 있다.
- GitHub push 후 Streamlit Cloud에서 자동 배포가 가능하다.
- 테스트 버전은 별도 데이터베이스 없이 샘플 데이터로 실행 가능하다.

## 8. 비기능 요구사항

### 8.1 성능

- 팀원 50명, 프로젝트 100개, 일정 5,000건, 채팅 메시지 50,000건 수준에서 사용 가능해야 한다.
- 주요 목록 조회는 3초 이내 표시를 목표로 한다.
- 대량 데이터는 페이지네이션 또는 기간 필터를 적용한다.

### 8.2 보안

- DB 비밀번호와 API 키는 GitHub에 커밋하지 않는다.
- `.streamlit/secrets.toml`은 로컬 개발용으로만 사용하고 `.gitignore`에 포함한다.
- 배포 환경의 secrets 관리 기능을 사용한다.
- 비밀번호는 해시 저장한다.
- 사용자 권한에 따라 접근 가능한 팀/프로젝트/일정을 제한한다.
- 테스트 버전의 샘플 계정은 실제 회사 계정이나 실제 비밀번호를 사용하지 않는다.

### 8.3 사용성

- 좌측 사이드바에 주요 메뉴를 배치한다.
- 모바일에서도 최소한 조회와 간단한 등록이 가능해야 한다.
- 일정 유형별 색상 체계를 유지한다.
- 목록 화면에서 검색/필터를 먼저 제공한다.

### 8.4 유지보수성

- 기능별 모듈 구조를 분리한다.
- 데이터 접근 코드는 별도 저장소 계층으로 분리한다.
- 테스트 버전은 JSON 저장소를 사용하고, 운영 버전은 DB 저장소로 교체 가능한 구조를 사용한다.
- UI 페이지와 비즈니스 로직을 최대한 분리한다.
- 환경 설정은 `.streamlit/secrets.toml` 또는 환경 변수로 관리한다.

### 9.2 테스트 버전 데이터 저장 계획

테스트 버전은 별도 데이터베이스 없이 동작해야 하므로 다음 방식을 사용한다.

- 앱 시작 시 `data/*.json` 파일에서 샘플 데이터를 로드한다.
- 화면에서 생성/수정/삭제한 데이터는 우선 `st.session_state`에 반영한다.
- 로컬 테스트에서는 JSON 파일 저장 기능을 제공할 수 있다.
- Streamlit Cloud에서는 파일 쓰기가 영구 저장을 보장하지 않으므로, 테스트 배포 버전은 샘플 데이터 기반 데모와 세션 내 변경을 기준으로 한다.
- 향후 운영 전환 시 `src/storage.py` 또는 `src/repositories.py` 내부 구현만 DB 방식으로 교체한다.

#### 테스트 데이터 파일 예시

```text
data/
├── users.json
├── teams.json
├── projects.json
├── project_members.json
├── tasks.json
├── schedules.json
├── chat_rooms.json
└── chat_messages.json
```

## 10. 데이터 모델 초안

테스트 버전에서는 아래 모델을 DB 테이블이 아니라 JSON 배열 구조로 저장한다. 운영 전환 시 동일한 필드를 DB 테이블로 이관한다.

### 10.1 users

| 컬럼 | 타입 | 설명 |
| --- | --- | --- |
| id | UUID/INT | 사용자 ID |
| email | TEXT | 로그인 이메일 |
| password_hash | TEXT | 비밀번호 해시 |
| name | TEXT | 이름 |
| role | TEXT | Admin/Manager/Member |
| team_id | FK | 소속 팀 |

## 14. 위험 요소 및 대응 방안

| 위험 요소 | 설명 | 대응 방안 |
| --- | --- | --- |
| Streamlit 실시간성 한계 | 완전한 실시간 채팅 구현이 어려움 | MVP는 주기적 새로고침 방식 사용 |
| 인증 보안 | 자체 인증 구현 시 보안 리스크 | Supabase Auth 또는 검증된 해시 라이브러리 사용 |
| 테스트 데이터 영구 저장 한계 | Streamlit Cloud의 파일 저장은 영구 저장 용도로 부적합 | 테스트 버전은 샘플 데이터와 세션 변경 중심으로 운영 |
| 운영 전환 비용 | JSON 저장소에서 DB로 전환 필요 | 저장소 계층을 분리해 UI 수정 없이 교체 |
| DB secrets 노출 | 운영 전환 시 GitHub에 민감 정보 커밋 위험 | `.gitignore`와 Streamlit Secrets 사용 |
| 권한 누락 | 다른 팀 데이터 노출 가능성 | 모든 데이터 조회에 사용자 권한 필터 적용 |
| 일정 UI 한계 | Streamlit 기본 위젯만으로 캘린더 구현 제한 | 초기에는 표/필터 기반, 이후 캘린더 컴포넌트 도입 |

## 17. 오픈 이슈

- 회사 내부 계정 체계와 연동할지 여부
- 휴가/출장 승인 기능을 MVP에 포함할지 여부
- 채팅을 Streamlit 내부 기능으로 유지할지, 외부 메신저 연동으로 확장할지 여부
- 파일 첨부 기능 필요 여부
- 모바일 사용 비중
- 사용자 수와 예상 데이터 규모

## 18. 결론

MVP 테스트 버전은 데이터베이스 없이 JSON 샘플 데이터와 `st.session_state`를 사용해 프로젝트, 일정, 채팅을 통합 관리하는 내부 협업 도구로 구현한다. 단, 실시간 채팅, 고급 캘린더, 대규모 권한 관리, 영구 저장은 Streamlit과 파일 기반 저장의 한계를 고려해 운영 전환 단계에서 DB 기반으로 확장한다.

초기 개발은 테스트 저장소, 인증, 사용자/팀, 프로젝트, 일정 기능을 먼저 완성하고, 이후 채팅과 대시보드를 붙이는 순서가 가장 현실적이다.
